# Security Rules

## Vulnerability Reports

Report vulnerabilities, exploits, leaked secrets, and security-sensitive bugs
through [GitHub Security Advisories](https://github.com/agentteamhq/agentteam-email/security/advisories/new),
not public issues, discussions, pull requests, or announcements. Do not publicly
disclose details until coordinated disclosure is complete.

## Approval

- Agents must read this file before security-sensitive work.
- Agents must obtain explicit current-task human approval before changing
  authentication, authorization, Better Auth configuration, credential handling,
  encryption, sessions, cookies, API keys, token behavior, JWT behavior, OAuth
  behavior, JWKS behavior, secret storage, or security-sensitive routes.
- Approval must be written in the current task and must include the exact phrase
  `I give alteration permission approval`.
- Approval applies only to the security-sensitive scope named in the current
  task.
- Agents must limit unapproved security-sensitive work to read-only review.

## Credential Boundaries

- Credentials must remain inside the boundary that owns their authentication,
  authorization, storage, or execution use.
- Cookies, raw API keys, OAuth tokens, JWTs, raw `Authorization` headers,
  passwords, decrypted secrets, service tokens, runtime tokens, signing keys,
  encryption keys, and secret-derived hashes must not be forwarded, returned,
  logged, exported, or copied into another subsystem unless that subsystem owns
  the specific credential use.
- Better Auth credentials must terminate at the Better Auth-owned boundary.
- Service credentials must identify the service principal that owns the
  operation.
- User credentials must identify the authenticated user or accepted user-owned
  credential class.

## Encrypted Values

- Encrypted values are write-and-use data.
- Decrypted values must be available only at the execution boundary that uses the
  value for its intended operation.
- UI, RPC, API, logs, exports, diagnostics, support tools, and admin surfaces
  must not return or render decrypted values.
- Read surfaces may return operational metadata only when that metadata cannot
  reconstruct the secret value.

## Authentication And Authorization

- Handlers that read, decrypt, write, use, or describe secret-bearing records
  must authenticate the caller and authorize the authenticated principal for the
  exact object and operation before accessing the secret-bearing record.
- Caller-supplied identifiers, names, labels, titles, and display strings are
  filters only.
- Authority must come from the authenticated principal and persisted
  authorization state.

## HTTP Semantics

- Missing, malformed, expired, wrong-audience, wrong-issuer, or invalid
  credentials must return `401 Unauthorized`.
- Bearer authentication failures must include the applicable
  `WWW-Authenticate` challenge.
- Valid credentials without required authority must return `403 Forbidden`.
- Request validation errors must not be reported as authentication or
  authorization failures.

## Security Defaults

- Security-sensitive behavior must fail closed when required credentials, keys,
  issuer values, audience values, authorization state, or secret inputs are
  missing or invalid.
- Credentials and tokens must use the narrowest scope and shortest lifetime that
  satisfies the boundary they authenticate.

## Logging And Audit

- Logs and audit records must not contain raw credentials, cookies, tokens,
  passwords, decrypted secrets, raw secret values, broad payload dumps, or raw
  external `Authorization` headers.
- Security-sensitive logs and audit records must contain stable correlation
  fields when those fields are available.
- Audit records for authentication, authorization, credential, session, cookie,
  API key, encryption, or secret-storage events must be generated by the server
  boundary that owns the event.

## Tests

- Security-sensitive changes must include tests for each changed security
  boundary. Tests must prove secret non-disclosure, credential-boundary
  preservation, authorization failure behavior, and `401`/`403` semantics for
  changes that touch those behaviors.
