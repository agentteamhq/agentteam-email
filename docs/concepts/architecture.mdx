AgentTeam Email is built around a single public boundary — the authenticated web server — with every other service kept strictly internal. Understanding this boundary helps you deploy safely, configure Cloudflare correctly, and reason about how mail enters and leaves the system.

## Public boundary

The web server is the only component that accepts external traffic. Every browser session, public API call, agent runtime request, and CLI connection enters through it. The web server handles authentication (Better Auth sessions, OAuth, API keys), authorization, credential management, Cloudflare integration setup, and the product UI.

The web server also owns the inbound worker notification endpoint. Your Cloudflare Email Worker sends a signed `POST /rpc/agent-mail/ingest/v1` to this endpoint after it has written an inbound message to archive storage. That route accepts only HMAC-signed Worker notifications — it is not an admin, operator, or direct mail-control surface.

<Warning>
  Internal services — WildDuck, Haraka, ZoneMTA, Rspamd, MongoDB, Redis, and the mail control service — must
  never be reachable from the public internet. Exposing any internal service bypasses AgentTeam Email's
  authorization layer and the security boundary that protects mailbox contents and operator credentials.
</Warning>

## Cloudflare Email Routing and Worker

Inbound mail for your domain arrives at the Cloudflare edge. You configure Cloudflare Email Routing to forward all mail for your domain to your Cloudflare Email Worker. The Worker is the edge component you deploy as part of the AgentTeam Email setup.

When a message arrives, the Worker:

1. Writes the raw RFC822 bytes (`raw.eml`) to your R2-compatible archive bucket.
2. Writes envelope and authentication metadata (`edge.json`) as the durable inbound commit marker.
3. Sends a single HMAC-signed notification to your deployment's `/rpc/agent-mail/ingest/v1` endpoint.

The Worker never sends raw mail bytes to your deployment — only a signed notification with bundle metadata. The archive is the source of truth; the notification is the default ingest signal, and the periodic reconciler sweep is the recovery path.

## R2-compatible archive storage

AgentTeam Email requires an R2-compatible object storage bucket as a durable archive for all inbound and outbound mail. For self-hosted installs you supply your own bucket and credentials. The bucket holds raw inbound messages, edge metadata, relay records, and delivery results, organized by organization, domain, date, and message ID.

## Internal mail delivery pipeline

The internal mail runtime handles inbound replay, outbound delivery, spam scoring, bounce processing, and feedback routing. It consists of:

- **WildDuck** — IMAP server and mailbox storage for all agent mailboxes.
- **Haraka** — inbound SMTP delivery into WildDuck after archive replay.
- **ZoneMTA** — outbound SMTP queue, retries, and bounce generation.
- **Rspamd** — spam scoring integrated with the inbound pipeline.
- **Mail control service** — coordinates inbound replay, outbound relay, provisioning, feedback processing, and status.
- **MongoDB and Redis** — state, queuing, and caching for the services above.

You choose the full stack through your Compose or Helm configuration. You do not configure these services directly — they are wired together by the deployment and controlled through the web server and mail control service.

## Outbound providers

The mail control service routes outbound messages through one of two supported providers, selected by the `AGENT_MAIL_OUTBOUND_PROVIDER` configuration value:

- **`cloudflare`** — Cloudflare Email Sending.
- **`ses`** — Amazon SES.

Messages addressed to active local AgentTeam Email domains are routed back through Haraka and WildDuck without calling the outbound provider.

## Network boundaries

| Traffic                                   | Path                                               |
| ----------------------------------------- | -------------------------------------------------- |
| Browsers and API clients                  | Web server only                                    |
| Cloudflare Worker notifications           | `POST /rpc/agent-mail/ingest/v1` on the web server |
| CLI and agent mailbox clients             | Web server only                                    |
| Web server → internal services            | Internal network, server-side only                 |
| WildDuck, Haraka, ZoneMTA, MongoDB, Redis | Internal network, not reachable publicly           |

## Deployment shapes

AgentTeam Email supports two deployment shapes for self-hosted installs.

<CardGroup cols={2}>
  <Card
    title='Docker Compose'
    icon='docker'
    href='/guides/self-host/docker-compose'
  >
    Deploy all services on a single host using the root `compose.yaml`. The web server port is published; all
    other services stay on the Compose network.
  </Card>
  <Card
    title='Helm (Kubernetes)'
    icon='dharmachakra'
    href='/guides/self-host/helm'
  >
    Deploy to Kubernetes using the `charts/agentteam-email` Helm chart. The web service is exposed through
    optional Ingress; all internal services are cluster-internal.
  </Card>
</CardGroup>

Both shapes use fail-closed configuration — required values such as secrets, storage endpoints, and provider credentials must be supplied explicitly. The deployment will not start if required inputs are missing.

If your deployment is not directly reachable from the public internet, run your reverse proxy, Tailscale Funnel, Cloudflare Tunnel, or equivalent connector outside the AgentTeam Email Compose or Helm stack. That infrastructure should route `PUBLIC_HOSTNAME` to the web server; the internal mail control service and data stores remain private.
