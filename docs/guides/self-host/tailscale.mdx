Tailscale Funnel is one of two supported options for exposing AgentTeam Email to inbound Cloudflare Worker traffic. Funnel publishes two routes from your tailnet to the public internet — the web application and the fast-path ingest endpoint at `POST /agent-mail/ingest/v1` — so your deployed Cloudflare Worker can deliver signed notifications to the mail control service after archiving each inbound message to R2.

<Warning>
  Funnel must only expose the web application and the `/agent-mail/ingest/v1` route. Do **not** expose WildDuck, MongoDB, Redis, Haraka, ZoneMTA, or the internal control API through Funnel or any other public interface. Exposing these services creates direct attack surface against your mail store and internal API boundary.
</Warning>

## Environment Variables

Set the following variables before starting the Tailscale container or deploying your Helm release.

```text
AGENT_MAIL_CF_TUNNEL_LISTEN_URL=http://0.0.0.0:8080
AGENT_MAIL_CF_TUNNEL_EXTERNAL_URL=https://mail-ingress.company.example/agent-mail/ingest/v1
AGENT_MAIL_CF_TUNNEL_HMAC_SECRET=<random-secret>
TAILSCALE_AUTHKEY=
TAILSCALE_HOSTNAME=mail-ingress
```

`AGENT_MAIL_CF_TUNNEL_HMAC_SECRET` must match the secret bound to the Cloudflare Worker. Requests signed with a different secret or with no signature are rejected before they reach the mail control service.

## Helm Values

If you are deploying to Kubernetes, configure the tunnel and Tailscale blocks in your Helm values file. Reference the HMAC secret and Tailscale auth key from a Kubernetes Secret rather than embedding them as plain values.

```yaml
tunnel:
  provider: tailscale
  listenUrl:
    value: http://0.0.0.0:8080
  externalUrl:
    value: https://mail-ingress.company.example/agent-mail/ingest/v1
  hmacSecret:
    valueFrom:
      secretKeyRef:
        name: mail-secrets
        key: AGENT_MAIL_CF_TUNNEL_HMAC_SECRET

tailscale:
  hostname: mail-ingress
  certDomain: mail-ingress.company.example
  authKey:
    valueFrom:
      secretKeyRef:
        name: mail-secrets
        key: AGENT_MAIL_TAILSCALE_AUTH_KEY
```

## Docker Compose

The Tailscale container shares the network namespace of the web server container via `network_mode: "service:atemail-web-server"`, which means `127.0.0.1:4321` resolves to the web server listener. Funnel configuration is driven by the `TS_SERVE_CONFIG` file mounted at `/etc/tailscale/serve/serve.json`.

```yaml
services:
  tailscale:
    image: ghcr.io/tailscale/tailscale:v1.96.5
    network_mode: "service:atemail-web-server"
    environment:
      TS_AUTHKEY: "${TS_AUTHKEY:?missing TS_AUTHKEY}"
      TS_HOSTNAME: "${TS_HOSTNAME:-mail-ingress}"
      TS_STATE_DIR: /var/lib/tailscale
      TS_SERVE_CONFIG: /etc/tailscale/serve/serve.json
      TS_USERSPACE: "true"
      TS_AUTH_ONCE: "true"
    volumes:
      - tailscale-state:/var/lib/tailscale
      - ./tailscale:/etc/tailscale/serve:ro
    depends_on:
      - atemail-web-server
    restart: unless-stopped

volumes:
  tailscale-state:
```

Create `./tailscale/serve.json` to forward port 443 to the web server and enable Funnel for the tunnel hostname:

```json
{
  "TCP": {
    "443": {
      "TCPForward": "127.0.0.1:4321",
      "TerminateTLS": "mail-ingress.example.ts.net"
    }
  },
  "AllowFunnel": {
    "mail-ingress.example.ts.net:443": true
  }
}
```

## Kubernetes

The Helm chart does not install or configure Tailscale. Run Tailscale as operator-owned infrastructure outside the chart and point Funnel at the web server Kubernetes service:

```text
http://atemail-web-server:80
```

If the Tailscale workload runs in a different namespace than the AgentTeam Email release, use the fully qualified Kubernetes DNS name for the `atemail-web-server` service.

## Setup Flow

<Steps>
  <Step title="Create a Tailscale auth key">
    In the Tailscale admin console, go to **Settings → Keys** and generate a new auth key. Use a reusable key if you plan to restart the container, or an ephemeral key for a short-lived test. Store the value in `TAILSCALE_AUTHKEY` (Compose) or the `mail-secrets` Kubernetes Secret.
  </Step>

  <Step title="Configure the Tailscale hostname">
    Set `TAILSCALE_HOSTNAME` to `mail-ingress` or another hostname that is unique within your tailnet. This becomes the subdomain of your tailnet DNS name, for example `mail-ingress.example.ts.net`.
  </Step>

  <Step title="Configure HTTPS serving for your tunnel hostname">
    For Docker Compose, create `./tailscale/serve.json` with the `TCP`, `TerminateTLS`, and `AllowFunnel` blocks shown above. For Kubernetes, configure your Tailscale operator or sidecar to serve HTTPS and enable Funnel for `mail-ingress.company.example`.
  </Step>

  <Step title="Set the external URL and HMAC secret">
    Set `AGENT_MAIL_CF_TUNNEL_EXTERNAL_URL` to the full public HTTPS ingest URL and `AGENT_MAIL_CF_TUNNEL_HMAC_SECRET` to a randomly generated secret:

    ```text
    AGENT_MAIL_CF_TUNNEL_EXTERNAL_URL=https://mail-ingress.company.example/agent-mail/ingest/v1
    AGENT_MAIL_CF_TUNNEL_HMAC_SECRET=<random-secret>
    ```
  </Step>

  <Step title="Apply your Helm release or start Docker Compose">
    Deploy with `helm upgrade --install` or `docker compose up -d`. The Tailscale container authenticates to your tailnet, acquires the configured hostname, and begins serving HTTPS traffic via Funnel.
  </Step>

  <Step title="Confirm both routes are reachable">
    Verify the web application is accessible at your public hostname and that the ingest endpoint responds (with a rejection, as expected for an unsigned request):

    ```bash
    curl -i https://mail-ingress.company.example/agent-mail/ingest/v1
    ```

    A `401` or `403` response confirms the route is live and the signature check is active. A connection error indicates Funnel is not yet propagated.
  </Step>

  <Step title="Deploy or update the Cloudflare Worker">
    Deploy the `agent-mail-ingress` Worker with `AGENT_MAIL_CF_TUNNEL_EXTERNAL_URL` and `AGENT_MAIL_CF_TUNNEL_HMAC_SECRET` set to the same values you configured in the previous steps. See [Cloudflare Authentication](/guides/self-host/cloudflare-authentication) for the full Worker setup flow.
  </Step>

  <Step title="Send a test email and confirm delivery">
    Send an email to any address at your inbound mail domain. Check your mail control service logs to confirm the Worker notification was accepted by the fast-path gate and the message was delivered to the correct mailbox.
  </Step>
</Steps>

## Verification

Check Funnel status from within the running Tailscale container:

```bash
docker compose exec tailscale tailscale serve status --json
```

The expected output for a Docker Compose deployment is:

```json
{
  "TCP": {
    "443": {
      "TCPForward": "127.0.0.1:4321",
      "TerminateTLS": "mail-ingress.example.ts.net"
    }
  },
  "AllowFunnel": {
    "mail-ingress.example.ts.net:443": true
  }
}
```

For Kubernetes, the `TCPForward` target should reference the Kubernetes service instead:

```json
{
  "TCP": {
    "443": {
      "TCPForward": "atemail-web-server:80",
      "TerminateTLS": "mail-ingress.example.ts.net"
    }
  },
  "AllowFunnel": {
    "mail-ingress.example.ts.net:443": true
  }
}
```

To confirm the ingest endpoint is live and the signature gate is active, send an unsigned `POST` and expect a rejection:

```bash
curl -i https://mail-ingress.company.example/agent-mail/ingest/v1
```

<Tip>
  An unsigned request should always be rejected. If it succeeds, the HMAC middleware is not running and your ingest endpoint is unprotected.
</Tip>
