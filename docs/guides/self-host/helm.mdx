The Helm chart is the supported Kubernetes deployment path for AgentTeam Email. It renders the full stack — web server, mail control service, MongoDB, Redis, WildDuck, Haraka, ZoneMTA, and Rspamd — into your cluster from a single OCI chart reference. Operator-specific choices live entirely in a release-owned `values.yaml`; the chart owns stable in-cluster defaults.

<Note>
  **Before you begin, make sure you have:**
  - A running Kubernetes cluster with a default StorageClass
  - Helm 3 installed (`helm version`)
  - A domain using Cloudflare DNS with Email Routing available
  - An R2-compatible object storage bucket and access credentials
  - A publicly routable hostname for the web app (for example, `https://mail.company.example`)
</Note>

## Chart reference

The chart is published to GHCR as an OCI artifact:

```text
oci://ghcr.io/agentteamhq/agentteam-email
```

No Helm repository add step is required — Helm 3.8+ can pull OCI charts directly.

## Steps

<Steps>
  <Step title="Create your values file">
    Create a `helm/values.yaml` file in your working directory. This file is your release configuration — keep it in version control, but keep secrets out of it using the `valueFrom` pattern described below.

    Start from the minimal required values (see the [Key values](#key-values) section below for a full annotated example).
  </Step>

  <Step title="Render to validate">
    Render the chart locally before installing to catch configuration errors early:

    ```bash
    helm template atemail oci://ghcr.io/agentteamhq/agentteam-email \
      --namespace agentteam-email \
      -f helm/values.yaml
    ```

    Review the rendered manifests to confirm secrets, hostnames, and ingress rules look correct.
  </Step>

  <Step title="Install or upgrade">
    Install the chart or upgrade an existing release with a single command:

    ```bash
    helm upgrade --install atemail oci://ghcr.io/agentteamhq/agentteam-email \
      --namespace agentteam-email \
      --create-namespace \
      -f helm/values.yaml
    ```

    Omitting `--version` installs the latest stable chart. Pin a specific chart version for reproducible production rollouts by adding `--version 0.5.0`.
  </Step>

  <Step title="Verify pods are running">
    Wait for the main workloads to become ready:

    ```bash
    kubectl get pods -n agentteam-email
    ```

    For a more thorough readiness check, wait on individual rollouts:

    ```bash
    kubectl rollout status deployment/atemail-web-server -n agentteam-email --timeout=180s
    kubectl rollout status deployment/atemail-mail-control-service -n agentteam-email --timeout=180s
    ```

    Once both are ready, open `publicHostname` in your browser and sign in.
  </Step>
</Steps>

## Key values

The chart values are semantic service settings, not a raw environment-variable bag. You provide only the deployment-specific values; the chart handles stable defaults such as in-cluster DNS names, container ports, and service config.

The following annotated snippet covers all required operator values:

```yaml
namespace:
  create: true
  name: agentteam-email

# Public URL of the web app. The Cloudflare Worker derives the ingest URL
# by appending /agent-mail/ingest/v1 to this value.
publicHostname: https://mail.company.example

webServer:
  # Random secret for session signing. Use openssl rand -base64 32.
  authSecret:
    value: '<random-secret>'
  # 32-byte key for encrypting stored Worker secrets (base64url-encoded).
  encryptionKey:
    value: '<base64url-32-byte-key>'

controlApi:
  # Shared token for the web-to-control-service API boundary.
  token:
    value: '<random-control-token>'

wildduck:
  adminAccessToken:
    value: '<random-wildduck-token>'
  accessControlSecret:
    value: '<random-wildduck-secret>'

mailRuntime:
  loopSecret:
    value: '<random-mail-loop-secret>'
  zonemtaRelayPassword:
    value: '<random-zonemta-relay-password>'
  feedbackMailboxPassword:
    value: '<random-feedback-mailbox-password>'

objectStorage:
  endpoint:
    value: 'https://<account-id>.r2.cloudflarestorage.com'
  region:
    value: auto
  bucket:
    value: company-agent-mail-archive
  accessKeyId:
    value: '<r2-access-key>'
  secretAccessKey:
    value: '<r2-secret-key>'

cloudflare:
  apiBaseUrl: https://api.cloudflare.com/client/v4
  accountId:
    value: '<cloudflare-account-id>'
  apiToken:
    value: '<cloudflare-token>'
  worker:
    scriptName: agent-mail-ingress
    archiveBucket:
      value: company-agent-mail-archive

outbound:
  # Outbound provider: cloudflare or ses
  provider:
    value: cloudflare
```

### Referencing Kubernetes Secrets

Most sensitive values support a `valueFrom` pattern instead of an inline `value`. Use this in production so secrets stay in Kubernetes Secrets rather than in your values file:

```yaml
webServer:
  authSecret:
    valueFrom:
      secretKeyRef:
        name: mail-secrets
        key: BETTER_AUTH_SECRET
  encryptionKey:
    valueFrom:
      secretKeyRef:
        name: mail-secrets
        key: ENCRYPT_SECRET_KEY
```

The same pattern works for any value that accepts `valueFrom`:

```yaml
objectStorage:
  accessKeyId:
    valueFrom:
      secretKeyRef:
        name: mail-secrets
        key: AGENT_MAIL_R2_ACCESS_KEY_ID
  secretAccessKey:
    valueFrom:
      secretKeyRef:
        name: mail-secrets
        key: AGENT_MAIL_R2_SECRET_ACCESS_KEY
```

<Note>
  Secrets rendered into bundled service config files (Haraka, WildDuck, ZoneMTA) must use inline `mailRuntime.*.value` entries. The `valueFrom` pattern is not supported for those values because the chart renders them into ConfigMaps at install time.
</Note>

Ready-to-use example values files are available in `docs/examples/helm/` in the repository:

- `docs/examples/helm/values-basic.yaml` — inline values, suitable for initial setup
- `docs/examples/helm/values-existing-secret.yaml` — `valueFrom` pattern for all sensitive values

## Tunnel modes

The chart does not install or configure ingress tunnel software. You run tunnel infrastructure as operator-owned workloads and point it at the web server service. Choose the mode that matches your networking setup.

<Tabs>
  <Tab title="No tunnel (Ingress)">
    Use a standard Kubernetes Ingress when your cluster has a public load balancer or ingress controller. Enable and configure the chart's built-in ingress:

    ```yaml
    ingress:
      enabled: true
      className: nginx
      hosts:
        - host: mail.company.example
          paths:
            - path: /
              pathType: Prefix
      tls:
        - hosts:
            - mail.company.example
          secretName: mail-company-example-tls
    ```

    The web server service listens on port 80 inside the cluster. Your ingress controller forwards traffic to it.
  </Tab>
  <Tab title="Tailscale Funnel">
    Run Tailscale as operator-owned infrastructure alongside the chart. Point Funnel at the web server service in the release namespace:

    ```yaml
    # Your operator-owned Tailscale workload should forward to:
    # http://atemail-web-server:80
    #
    # Set publicHostname to your Funnel hostname:
    publicHostname: https://mail-ingress.example.ts.net
    ```

    If the Tailscale workload runs outside the `agentteam-email` namespace, use the fully-qualified service DNS name: `http://atemail-web-server.agentteam-email.svc.cluster.local:80`.

    See the [Tailscale guide](/guides/self-host/tailscale) for the full Funnel configuration.
  </Tab>
  <Tab title="Cloudflare Tunnel">
    Run `cloudflared` as operator-owned infrastructure and configure public hostname routes to forward traffic to the web server service:

    ```yaml
    # Your cloudflared tunnel should forward:
    # mail.company.example -> http://atemail-web-server:80
    #
    # Set publicHostname to match your tunnel's public hostname:
    publicHostname: https://mail.company.example
    ```

    Do not expose the internal control API, WildDuck, MongoDB, Redis, Haraka, ZoneMTA, or Rspamd through the tunnel.

    See the [Cloudflare Tunnel guide](/guides/self-host/cloudflared) for the full configuration.
  </Tab>
</Tabs>

## Persistence

The chart renders `PersistentVolumeClaim` resources only — no `hostPath` volumes, no static `PersistentVolume` resources, and no node selector assumptions. When `persistence.storageClassName` is empty, the chart omits `storageClassName` so your cluster's default StorageClass provisions the volumes automatically.

```yaml
persistence:
  enabled: true
  storageClassName: ''   # Leave empty to use the cluster default StorageClass

  mongodb:
    size: 20Gi
    accessModes: ['ReadWriteOnce']
    existingClaim: ''   # Set to use a pre-existing PVC
  redis:
    size: 2Gi
    accessModes: ['ReadWriteOnce']
    existingClaim: ''
  rspamd:
    size: 1Gi
    accessModes: ['ReadWriteOnce']
    existingClaim: ''
  harakaQueue:
    size: 5Gi
    accessModes: ['ReadWriteOnce']
    existingClaim: ''
```

Set `existingClaim` to the name of a pre-existing PVC when your environment manages storage independently. The chart mounts the named claim and skips PVC creation for that service.

<Warning>
  Setting `persistence.enabled=false` switches MongoDB, Redis, Rspamd, and the Haraka queue to `emptyDir` volumes. All data is lost when pods restart. Use this only for disposable development or test installs — never for production.
</Warning>

## Example values files

The repository ships ready-to-use example values files in `docs/examples/helm/`:

- **`values-basic.yaml`** — all required values using inline `value` entries; good for a first install

Reference these as starting points and customize for your environment.
