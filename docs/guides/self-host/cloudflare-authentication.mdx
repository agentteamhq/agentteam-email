Cloudflare handles four concerns for inbound mail in a self-hosted AgentTeam Email deployment: DNS for your mail domain, Email Routing to deliver inbound messages to the Worker, Worker deployment to archive messages and fire worker notifications, and R2 bucket storage for raw message archives. Each concern requires its own set of Cloudflare resources and a correctly scoped API token before you can deploy the Worker and enable mail flow.

## API Token Requirements

Create a Cloudflare API token scoped to one account and the zones that receive mail. The token must allow the setup workflow to perform each of the following actions.

- Read account and zone metadata
- Create or update the Worker script (`agent-mail-ingress`)
- Bind the Worker to an R2 bucket
- Bind Worker secrets
- Read and update Email Routing settings
- Read and update DNS records required by Email Routing
- Create or update Email Routing catch-all routes

<Warning>
  Never expose this token to browsers. Store it exclusively in server-side environment variables or a
  Kubernetes Secret. Any client-side exposure allows an attacker to modify your Worker, routing rules, and DNS
  records.
</Warning>

## Required Cloudflare Resources

Before running the setup workflow, create or identify each of the following resources in your Cloudflare account.

| Resource                | Example value                                         |
| ----------------------- | ----------------------------------------------------- |
| Cloudflare Account ID   | Your account identifier from the Cloudflare dashboard |
| Zone for inbound mail   | `company.example`                                     |
| R2 bucket name          | `company-agent-mail-archive`                          |
| Worker script name      | `agent-mail-ingress`                                  |
| Public web app hostname | `mail.company.example`                                |

## Worker Bindings

The `wrangler.toml` for the `agent-mail-ingress` Worker declares a `send_email` binding named `EMAIL`. The setup workflow provisions the remaining bindings as Worker secrets and environment variables.

```text
AGENTTEAM_ORGANIZATION_ID=<web-owned-organization-id>
AGENTTEAM_ORG_PUBLIC_ID=<web-owned-organization-public-id>
AGENTTEAM_CONNECTION_ID=<web-owned-connection-id>
AGENTTEAM_DOMAIN_ID=<web-owned-domain-id>
AGENTTEAM_DOMAIN=company.example
AGENTTEAM_ARCHIVE_PREFIX=orgs/<org-public-id>/domains/company.example/mail/inbound
AGENTTEAM_R2_ENDPOINT=<r2-endpoint>
AGENTTEAM_R2_BUCKET=company-agent-mail-archive
AGENTTEAM_R2_REGION=auto
AGENTTEAM_R2_ACCESS_KEY_ID=<temporary-r2-access-key>
AGENTTEAM_R2_SECRET_ACCESS_KEY=<temporary-r2-secret-key>
AGENTTEAM_R2_SESSION_TOKEN=<temporary-r2-session-token>
AGENTTEAM_R2_CREDENTIAL_EXPIRES_AT=<iso-timestamp>
AGENTTEAM_INGEST_URL=https://mail.company.example/rpc/agent-mail/ingest/v1
AGENTTEAM_WORKER_HMAC_SECRET=<per-connection-secret>
EMAIL=<Cloudflare send_email binding>
```

The web app provisions `AGENTTEAM_INGEST_URL` from `PUBLIC_HOSTNAME` and stores one encrypted HMAC secret per connected Worker connection. Operators do not set a global ingest URL or global Worker HMAC secret in Compose, Helm, or the mail control service.

## Email Routing Catch-All Route

Email Routing uses one enabled catch-all route per inbound mail zone. All domain-specific mailbox handling happens after the message reaches WildDuck and the web application's mail permissions model. Store the route configuration as explicit data so the setup workflow can apply it deterministically.

```json
{
  "zones": [
    {
      "zone_name": "company.example"
    }
  ],
  "routes": [
    {
      "enabled": true,
      "mode": "catch-all",
      "zone_name": "company.example",
      "address": "inbound@company.example"
    }
  ]
}
```

## Environment Variables

Set the following variables in your server-side environment, `.env` file, or Kubernetes Secret before running the setup workflow.

```text
AGENT_MAIL_CLOUDFLARE_API_TOKEN=
AGENT_MAIL_CLOUDFLARE_ACCOUNT_ID=
AGENT_MAIL_CLOUDFLARE_API_BASE_URL=https://api.cloudflare.com/client/v4
AGENT_MAIL_CLOUDFLARE_WORKER_SCRIPT_NAME=agent-mail-ingress
AGENT_MAIL_CLOUDFLARE_ROUTE_CONFIG=./config/email-routing.json

AGENT_MAIL_R2_ENDPOINT=
AGENT_MAIL_R2_REGION=auto
AGENT_MAIL_R2_BUCKET=company-agent-mail-archive
AGENT_MAIL_R2_ACCESS_KEY_ID=
AGENT_MAIL_R2_SECRET_ACCESS_KEY=
```

## Setup Flow

Complete the following steps in order. `PUBLIC_HOSTNAME` must reach the web server before Worker notifications can be delivered. If your server is not directly public, use operator-owned networking such as a reverse proxy, Tailscale Funnel, or Cloudflare Tunnel to route that hostname to the web server.

<Steps>
  <Step title="Create the R2 bucket">
    In the Cloudflare dashboard, navigate to **R2** and create a new bucket named `company-agent-mail-archive`. Note the bucket endpoint URL — you will need it for `AGENT_MAIL_R2_ENDPOINT`.
  </Step>

<Step title='Create the Cloudflare API token'>
  Go to **My Profile → API Tokens → Create Token**. Scope the token to your account and the zones that receive
  inbound mail. Grant every permission listed in the [API Token Requirements](#api-token-requirements) section
  above. Copy the token value and store it in `AGENT_MAIL_CLOUDFLARE_API_TOKEN`.
</Step>

  <Step title="Expose PUBLIC_HOSTNAME to the web server">
    Configure DNS, a reverse proxy, or operator-owned tunnel infrastructure so that `https://mail.company.example` reaches the AgentTeam Email web server. The same hostname must serve the web app and the Worker ingest endpoint:

    ```text
    https://mail.company.example/
    https://mail.company.example/rpc/agent-mail/ingest/v1
    ```

  </Step>

  <Step title="Set PUBLIC_HOSTNAME">
    Set `PUBLIC_HOSTNAME` to the public web server origin. Do not include the ingest path.

    ```text
    PUBLIC_HOSTNAME=https://mail.company.example
    ```

  </Step>

<Step title='Deploy the Worker'>
  Run the authenticated setup workflow in the web app to deploy the `agent-mail-ingress` Worker to your
  Cloudflare account. The workflow uses the Cloudflare token to upload the script, attach the R2 bucket
  binding, set `AGENTTEAM_INGEST_URL`, and generate the deployment-owned Worker HMAC secret.
</Step>

<Step title='Apply the Email Routing catch-all route'>
  Run the routing configuration step for each inbound zone. The workflow reads
  `AGENT_MAIL_CLOUDFLARE_ROUTE_CONFIG`, applies the catch-all route, and enables Email Routing for
  `company.example`.
</Step>

<Step title='Send a test email'>
  Send an email to any address at your inbound mail domain, for example `test@company.example`. The Worker
  receives the message via Email Routing and archives it to R2.
</Step>

  <Step title="Verify the R2 archive">
    Open the `company-agent-mail-archive` bucket in the Cloudflare dashboard and confirm that the inbound message produced three objects:

    - `raw.eml` — the original RFC 5322 message
    - `edge.json` — envelope metadata written by the Worker
    - `result.json` — delivery outcome written by the Worker

  </Step>

  <Step title="Verify worker notification delivery">
    Check your web server and mail control service logs and confirm that the signed Worker notification was accepted by the web RPC ingest route and that the message was delivered to the correct mailbox. A rejected notification indicates an unreachable `PUBLIC_HOSTNAME`, a stale Worker deployment, or a mismatched Worker HMAC secret.
  </Step>
</Steps>
