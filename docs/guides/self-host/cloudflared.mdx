Cloudflare Tunnel (`cloudflared`) is an alternative to Tailscale for exposing AgentTeam Email to inbound Cloudflare Worker traffic. The tunnel publishes two routes to the public internet — the web application and the fast-path ingest endpoint at `POST /agent-mail/ingest/v1` — so your deployed Worker can deliver signed notifications to the mail control service after archiving each inbound message to R2. Because the tunnel connector runs inside your infrastructure and makes outbound connections to Cloudflare, you do not need to open any inbound firewall ports.

<Warning>
  The tunnel must only expose the web application and the `/agent-mail/ingest/v1` route. Do **not** expose WildDuck, MongoDB, Redis, Haraka, ZoneMTA, Rspamd, or the internal control API. Exposing these services creates direct attack surface against your mail store, spam filter, and internal API boundary.
</Warning>

## DNS Setup

Create a Cloudflare Tunnel in the Cloudflare Zero Trust dashboard and configure a public hostname route that maps `mail-ingress.company.example` to your `cloudflared` connector. The connector forwards requests to the web server service inside your infrastructure. Cloudflare handles TLS termination and DNS automatically for tunnels attached to zones you manage.

For Docker Compose, the target is:

```text
http://atemail-web-server:4321
```

For Kubernetes, the target is:

```text
http://atemail-web-server:80
```

## Environment Variables

Set the following variables before starting the `cloudflared` container or deploying your Helm release.

```text
AGENT_MAIL_CF_TUNNEL_LISTEN_URL=http://0.0.0.0:8080
AGENT_MAIL_CF_TUNNEL_EXTERNAL_URL=https://mail-ingress.company.example/agent-mail/ingest/v1
AGENT_MAIL_CF_TUNNEL_HMAC_SECRET=<random-secret>
CLOUDFLARED_TUNNEL_TOKEN=
```

`AGENT_MAIL_CF_TUNNEL_HMAC_SECRET` must match the secret bound to the Cloudflare Worker. Requests signed with a different secret or with no signature are rejected before they reach the mail control service.

## Helm Values

If you are deploying to Kubernetes, configure the tunnel and `cloudflared` blocks in your Helm values file. Reference the HMAC secret and tunnel token from a Kubernetes Secret rather than embedding them as plain values.

```yaml
tunnel:
  provider: cloudflared
  listenUrl:
    value: http://0.0.0.0:8080
  externalUrl:
    value: https://mail-ingress.company.example/agent-mail/ingest/v1
  hmacSecret:
    valueFrom:
      secretKeyRef:
        name: mail-secrets
        key: AGENT_MAIL_CF_TUNNEL_HMAC_SECRET

cloudflared:
  tunnelToken:
    valueFrom:
      secretKeyRef:
        name: mail-secrets
        key: CLOUDFLARED_TUNNEL_TOKEN
```

## Docker Compose

The Compose stack does not configure Cloudflare Tunnel. Run `cloudflared` as operator-owned infrastructure on the same Docker network as the web server. Cloudflare owns all hostname and path routing for token-run tunnels; the connector only needs to start the tunnel.

```yaml
services:
  cloudflared:
    image: docker.io/cloudflare/cloudflared:2026.6.1
    command:
      - tunnel
      - --no-autoupdate
      - run
    environment:
      TUNNEL_TOKEN: "${TUNNEL_TOKEN:?missing TUNNEL_TOKEN}"
    restart: unless-stopped
```

## Kubernetes

The Helm chart does not install or configure `cloudflared`. Run Cloudflare Tunnel as operator-owned infrastructure outside the chart and point the public hostname routes at the web server Kubernetes service:

```text
http://atemail-web-server:80
```

If the `cloudflared` workload runs in a different namespace than the AgentTeam Email release, use the fully qualified Kubernetes DNS name for the `atemail-web-server` service.

## Setup Flow

<Steps>
  <Step title="Create a Cloudflare Tunnel">
    In the Cloudflare Zero Trust dashboard, go to **Networks → Tunnels → Create a tunnel**. Select **Cloudflared** as the connector type and give the tunnel a name such as `agent-mail-ingress`. Copy the tunnel token — you will store it as `CLOUDFLARED_TUNNEL_TOKEN`.
  </Step>

  <Step title="Create the DNS route for the tunnel hostname">
    In the tunnel's **Public Hostnames** configuration, add a route for `mail-ingress.company.example` pointing to the web server service (`http://atemail-web-server:4321` for Compose or `http://atemail-web-server:80` for Kubernetes). Cloudflare automatically creates the DNS CNAME record for you.
  </Step>

  <Step title="Store the tunnel token in your Secret">
    Add the tunnel token to your `.env` file or Kubernetes Secret under the key `CLOUDFLARED_TUNNEL_TOKEN`. For Kubernetes, reference it from the `mail-secrets` Secret as shown in the Helm values example above.
  </Step>

  <Step title="Configure ingress routing">
    In the Cloudflare Zero Trust dashboard, verify that the public hostname route for `mail-ingress.company.example` forwards all traffic — including the `/agent-mail/ingest/v1` path — to the web server. Web application traffic and fast-path ingest traffic share the same hostname and the same upstream service; path-based routing to the correct handler happens inside the web server.
  </Step>

  <Step title="Set the external URL and HMAC secret">
    Set `AGENT_MAIL_CF_TUNNEL_EXTERNAL_URL` to the full public HTTPS ingest URL and `AGENT_MAIL_CF_TUNNEL_HMAC_SECRET` to a randomly generated secret:

    ```text
    AGENT_MAIL_CF_TUNNEL_EXTERNAL_URL=https://mail-ingress.company.example/agent-mail/ingest/v1
    AGENT_MAIL_CF_TUNNEL_HMAC_SECRET=<random-secret>
    ```
  </Step>

  <Step title="Apply your Helm release or start Docker Compose">
    Deploy with `helm upgrade --install` or add the `cloudflared` service to your `compose.yaml` and run `docker compose up -d`. The connector authenticates to Cloudflare using the tunnel token and begins serving traffic through the configured public hostname routes.
  </Step>

  <Step title="Deploy or update the Cloudflare Worker">
    Deploy the `agent-mail-ingress` Worker with `AGENT_MAIL_CF_TUNNEL_EXTERNAL_URL` and `AGENT_MAIL_CF_TUNNEL_HMAC_SECRET` set to the same values you configured above. See [Cloudflare Authentication](/guides/self-host/cloudflare-authentication) for the full Worker setup flow.
  </Step>

  <Step title="Send a test email and confirm delivery">
    Send an email to any address at your inbound mail domain. Check your mail control service logs to confirm the Worker notification was accepted by the fast-path gate and the message was delivered to the correct mailbox.
  </Step>
</Steps>

## Verification

Confirm both the web application and the ingest endpoint are reachable through the tunnel:

```bash
curl -i https://mail-ingress.company.example/agent-mail/ingest/v1
curl -i https://mail.company.example/
```

The web hostname should return your web application. An unsigned `POST` to the ingest path should be rejected with a `401` or `403` — this confirms the route is live and the HMAC signature gate is active.

<Tip>
  If the ingest endpoint returns a connection error rather than a rejection, the tunnel connector is not yet running or the DNS record has not propagated. Check the tunnel status in the Cloudflare Zero Trust dashboard under **Networks → Tunnels**.
</Tip>
