Cloudflare Tunnel (`cloudflared`) can expose `PUBLIC_HOSTNAME` when your server is not directly reachable from the public internet. This is an operator-owned networking layer. AgentTeam Email Compose and Helm do not install, configure, or require `cloudflared`.

The tunnel must route the public hostname to the web server. The web server owns both the browser app and the canonical Worker ingest endpoint at `POST /rpc/agent-mail/ingest/v1`.

<Warning>
  The tunnel must expose only the web server. Do **not** expose WildDuck, MongoDB, Redis, Haraka, ZoneMTA,
  Rspamd, or the internal control API. Exposing these services creates direct attack surface against your mail
  store, spam filter, and internal API boundary.
</Warning>

## DNS Setup

Create a Cloudflare Tunnel in the Cloudflare Zero Trust dashboard and configure a public hostname route that maps `mail.company.example` to your `cloudflared` connector. The connector forwards requests to the web server service inside your infrastructure. Cloudflare handles TLS termination and DNS automatically for tunnels attached to zones you manage.

For Docker Compose, the target is:

```text
http://atemail-web-server:4321
```

For Kubernetes, the target is:

```text
http://atemail-web-server:80
```

## Tunnel credential

Cloudflared needs only its own tunnel token. Store it outside AgentTeam Email's Compose and Helm configuration.

```text
CLOUDFLARED_TUNNEL_TOKEN=
```

## Docker Compose

The Compose stack does not configure Cloudflare Tunnel. Run `cloudflared` as operator-owned infrastructure on the same Docker network as the web server. Cloudflare owns hostname routing for token-run tunnels; the connector only needs to start the tunnel.

```yaml
services:
  cloudflared:
    image: docker.io/cloudflare/cloudflared:2026.6.1
    command:
      - tunnel
      - --no-autoupdate
      - run
    environment:
      TUNNEL_TOKEN: '${CLOUDFLARED_TUNNEL_TOKEN:?missing CLOUDFLARED_TUNNEL_TOKEN}'
    restart: unless-stopped
```

## Kubernetes

The Helm chart does not install or configure `cloudflared`. Run Cloudflare Tunnel as operator-owned infrastructure outside the chart and point the public hostname routes at the web server Kubernetes service:

```text
http://atemail-web-server:80
```

If the `cloudflared` workload runs in a different namespace than the AgentTeam Email release, use the fully qualified Kubernetes DNS name for the `atemail-web-server` service.

## Setup Flow

<Steps>
  <Step title="Create a Cloudflare Tunnel">
    In the Cloudflare Zero Trust dashboard, go to **Networks → Tunnels → Create a tunnel**. Select **Cloudflared** as the connector type and give the tunnel a name such as `agent-mail-ingress`. Copy the tunnel token — you will store it as `CLOUDFLARED_TUNNEL_TOKEN`.
  </Step>

<Step title='Create the DNS route for PUBLIC_HOSTNAME'>
  In the tunnel's **Public Hostnames** configuration, add a route for `mail.company.example` pointing to the
  web server service (`http://atemail-web-server:4321` for Compose or `http://atemail-web-server:80` for
  Kubernetes). Cloudflare automatically creates the DNS CNAME record for you.
</Step>

<Step title='Store the tunnel token in your Secret'>
  Store the tunnel token in the operator-owned secret or environment file that starts `cloudflared`. Do not
  add it to AgentTeam Email Helm values or Compose-required variables.
</Step>

<Step title='Configure ingress routing'>
  In the Cloudflare Zero Trust dashboard, verify that the public hostname route for `mail.company.example`
  forwards all traffic — including the `/rpc/agent-mail/ingest/v1` path — to the web server. Web application
  traffic and worker ingest traffic share the same hostname and the same upstream service; path-based
  routing to the correct handler happens inside the web server.
</Step>

<Step title='Apply your Helm release or start Docker Compose'>
  Deploy AgentTeam Email with `helm upgrade --install` or `docker compose up -d`. Start `cloudflared`
  separately as operator-owned infrastructure. The connector authenticates to Cloudflare using the tunnel
  token and begins serving traffic through the configured public hostname route.
</Step>

<Step title='Deploy or update the Cloudflare Worker'>
  Deploy the `agent-mail-ingress` Worker through the authenticated web setup flow. Backend provisioning
  derives `AGENTTEAM_INGEST_URL` from `PUBLIC_HOSTNAME` and generates the Worker HMAC secret. See [Cloudflare
  Authentication](/guides/self-host/cloudflare-authentication) for the full Worker setup flow.
</Step>

  <Step title="Send a test email and confirm delivery">
    Send an email to any address at your inbound mail domain. Check your web server and mail control service logs to confirm the Worker notification was accepted by the web RPC ingest route and the message was delivered to the correct mailbox.
  </Step>
</Steps>

## Verification

Confirm both the web application and the ingest endpoint are reachable through the tunnel:

```bash
curl -i https://mail.company.example/rpc/agent-mail/ingest/v1
curl -i https://mail.company.example/
```

The web hostname should return your web application. An unsigned `POST` to the ingest path should be rejected with a `401` or `403` — this confirms the route is live and the HMAC signature gate is active.

<Tip>
  If the ingest endpoint returns a connection error rather than a rejection, the tunnel connector is not yet
  running or the DNS record has not propagated. Check the tunnel status in the Cloudflare Zero Trust dashboard
  under **Networks → Tunnels**.
</Tip>
