The root `compose.yaml` is the supported single-host deployment path. It runs the web server, mail control service, MongoDB, Redis, WildDuck, Haraka, Rspamd, and ZoneMTA together on one Compose network. All services communicate over that internal network; only the web app is exposed publicly.

<Note>
  **Before you begin, make sure you have:**
  - Docker Engine and Docker Compose v2 (`docker compose version`)
  - A domain using Cloudflare DNS with Email Routing available
  - An R2-compatible object storage bucket and access credentials
  - A publicly routable hostname for the web app (for example, `https://mail.company.example`)
</Note>

## Steps

<Steps>
  <Step title="Clone the repository">
    Clone the AgentTeam Email repository and change into the project root:

    ```bash
    git clone https://github.com/agentteamhq/agentteam-email
    cd agentteam-email
    ```
  </Step>

  <Step title="Create your .env file">
    Copy the environment example file to `.env` in the project root:

    ```bash
    cp docs/examples/compose/.env.example .env
    ```

    Open `.env` in your editor and populate every required value. The example file contains placeholder values for all variables the stack needs. The Compose file uses fail-closed syntax (`${VAR:?missing VAR}`) for required values, so the stack will refuse to start if any are absent.
  </Step>

  <Step title="Configure required environment variables">
    Set the following variables in your `.env` file. Each group controls a distinct part of the stack.

    **Web app and authentication**

    | Variable | Description |
    |---|---|
    | `PUBLIC_HOSTNAME` | Public URL of the web app, e.g. `https://mail.company.example` |
    | `BETTER_AUTH_SECRET` | Random secret for session signing. Generate with `openssl rand -base64 32` |
    | `ENCRYPT_SECRET_KEY` | 32-byte key used to encrypt stored Worker secrets (base64url-encoded) |

    **Databases**

    | Variable | Description |
    |---|---|
    | `AGENTTEAM_EMAIL_APP_MONGODB_URI` | MongoDB connection URI for the web app database |
    | `AGENTTEAM_EMAIL_WILDDUCK_MONGODB_URI` | MongoDB connection URI for WildDuck |
    | `AGENTTEAM_EMAIL_CONTROL_MONGODB_URI` | MongoDB connection URI for the mail control service |
    | `AGENTTEAM_EMAIL_REDIS_URL` | Redis connection URL |

    The default `compose.yaml` provides bundled MongoDB and Redis containers. These variables default to those internal services, so you only need to set them explicitly when using external databases via `compose.no-db.yaml`.

    **Object storage (R2 or S3-compatible)**

    | Variable | Description |
    |---|---|
    | `AGENT_MAIL_R2_ENDPOINT` | S3-compatible endpoint URL, e.g. `https://<account-id>.r2.cloudflarestorage.com` |
    | `AGENT_MAIL_R2_BUCKET` | Bucket name for raw message archives |
    | `AGENT_MAIL_R2_ACCESS_KEY_ID` | Storage access key ID |
    | `AGENT_MAIL_R2_SECRET_ACCESS_KEY` | Storage secret access key |

    **Cloudflare**

    | Variable | Description |
    |---|---|
    | `AGENT_MAIL_CLOUDFLARE_ACCOUNT_ID` | Your Cloudflare account ID |
    | `AGENT_MAIL_CLOUDFLARE_API_TOKEN` | API token with Email Routing, Worker, R2, and DNS permissions |
    | `AGENT_MAIL_CLOUDFLARE_WORKER_ARCHIVE_BUCKET` | R2 bucket bound to the Cloudflare Email Worker |

    **Internal service credentials**

    | Variable | Description |
    |---|---|
    | `AGENT_MAIL_CONTROL_API_TOKEN` | Shared token for the web-to-control-service API boundary |
    | `AGENT_MAIL_WILDDUCK_ADMIN_ACCESS_TOKEN` | WildDuck admin API token |
    | `AGENT_MAIL_WILDDUCK_ACCESS_CONTROL_SECRET` | WildDuck access control secret |
    | `AGENT_MAIL_LOOP_SECRET` | Anti-loop secret for ZoneMTA SRS rewriting |
    | `AGENT_MAIL_ZONEMTA_RELAY_PASSWORD` | Password for ZoneMTA relay authentication |
    | `AGENT_MAIL_FEEDBACK_MAILBOX_PASSWORD` | Password for the feedback/bounce mailbox |

    **Outbound provider**

    | Variable | Description |
    |---|---|
    | `AGENT_MAIL_OUTBOUND_PROVIDER` | Outbound email provider: `cloudflare` or `ses` |

    When `AGENT_MAIL_OUTBOUND_PROVIDER=ses`, also set `AGENT_MAIL_AWS_REGION`, `AGENT_MAIL_AWS_ACCESS_KEY_ID`, and `AGENT_MAIL_AWS_SECRET_ACCESS_KEY`.

    <Tip>
      Generate strong random values for all secrets and tokens with `openssl rand -hex 32` or `openssl rand -base64 32`. Use a different value for each variable.
    </Tip>
  </Step>

  <Step title="Start the stack">
    Bring up all services in detached mode:

    ```bash
    docker compose up -d
    ```

    Check that all containers are running and healthy:

    ```bash
    docker compose ps
    ```

    Wait until the `atemail-web-server` and `atemail-mail-control-service` containers show a healthy status before proceeding. MongoDB, Redis, and WildDuck must be healthy before the application containers start.
  </Step>

  <Step title="Open the web app">
    Navigate to the URL you set as `PUBLIC_HOSTNAME` in your browser and sign in. From the web app you can create agent mailboxes, configure Cloudflare connections, and manage credentials.

    Complete your Cloudflare setup through the authenticated web surface to provision the Email Worker and activate inbound routing. See the [Cloudflare setup guide](/guides/self-host/cloudflare-authentication) for the full walkthrough.
  </Step>
</Steps>

## Container images

The two first-party images are published under the GHCR namespace:

```text
ghcr.io/agentteamhq/agentteam-email/atemail-web-server
ghcr.io/agentteamhq/agentteam-email/atemail-mail-control-service
```

**Version pinning**

By default, the stack pulls `latest`. Pin a specific release by setting `AGENTTEAM_EMAIL_VERSION` in your `.env`:

```dotenv
AGENTTEAM_EMAIL_VERSION=0.5.0
```

**Pull policy**

The default pull policy is `always`, which refreshes the selected tag on every `docker compose up`. Override it when working with a locally built or explicitly pinned image:

```dotenv
AGENTTEAM_EMAIL_PULL_POLICY=if-not-present
```

## Persistent volumes

The default `compose.yaml` creates four named volumes that persist data across container restarts and upgrades:

| Volume | Contents |
|---|---|
| `mongodb-data` | MongoDB data directory |
| `redis-data` | Redis persistence files |
| `rspamd-data` | Rspamd learning and statistical data |
| `haraka-queue` | Haraka inbound mail queue |

Back up these volumes before upgrading. Data is retained across `docker compose down` and `docker compose up` as long as you do not pass `--volumes`.

## No-DB variant

Use `compose.no-db.yaml` when you supply MongoDB and Redis from outside the Compose stack — for example, when using a managed database service or a shared cluster:

```bash
docker compose -f compose.no-db.yaml up -d
```

With the no-DB variant, set all three MongoDB URIs and the Redis URL explicitly in your `.env`:

```dotenv
AGENTTEAM_EMAIL_APP_MONGODB_URI=mongodb://user:password@mongodb.example:27017/agentteam_email?authSource=admin&tls=true
AGENTTEAM_EMAIL_WILDDUCK_MONGODB_URI=mongodb://user:password@mongodb.example:27017/wildduck?authSource=admin&tls=true
AGENTTEAM_EMAIL_CONTROL_MONGODB_URI=mongodb://user:password@mongodb.example:27017/agent_mail_control?authSource=admin&tls=true
AGENTTEAM_EMAIL_REDIS_URL=rediss://:password@redis.example:6379/3
```

The no-DB variant only declares `rspamd-data` and `haraka-queue` as named volumes.

<Warning>
  Never commit your `.env` file to version control. It contains secrets, credentials, and API tokens. Add `.env` to your `.gitignore` before making any commits.
</Warning>
