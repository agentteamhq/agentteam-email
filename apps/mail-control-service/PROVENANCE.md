# Agent Mail Provenance

This document defines the durable ID, header, object-key, result metadata, and
provider metadata surfaces for Agent Mail. Consult it before adding, removing,
renaming, or repurposing IDs, headers, result metadata, or provider metadata.
The required R2 bucket shape is defined in `R2-BUCKET-LAYOUT.md` and must stay
consistent with this document.

## Purpose

Provenance is Agent Mail's message chain-of-custody. The point is to preserve
the IDs and metadata that prove where each message came from, where it moved,
which service boundary handled it, and where it ended up. IDs created or
received at Cloudflare, R2, WildDuck, ZoneMTA, the internal SMTP relay, and
outbound providers must be carried forward instead of replaced when they are
needed to trace that movement.

## Inbound

The Cloudflare Email Worker is the inbound edge. Cloudflare Email Routing does
not expose a separate durable message ID in the Worker runtime API. Agent Mail's
UUIDv7 `ingest_id` is therefore the Cloudflare-boundary ID.

Inbound surfaces:

- `ingest_id`: UUIDv7 generated by the Worker for one inbound archive bundle.
- `raw_sha256`: SHA-256 of the exact raw RFC822 bytes received by the Worker.
- `organization_id` and `organization_public_id`: web-owned organization
  identity resolved before Worker archive writes.
- `archive_prefix`: the active domain archive prefix,
  `orgs/<org_public_id>/domains/<recipient_domain>/mail/inbound`.
- `worker_connection_id` and `worker_domain_deployment_id`: web-owned Worker
  connection and domain deployment identity validated before mail-control
  enqueue.
- Inbound raw R2 key:
  `orgs/<org_public_id>/domains/<recipient_domain>/mail/inbound/YYYY/MM/DD/<ingest_id>/raw.eml`.
- Inbound edge metadata R2 key:
  `orgs/<org_public_id>/domains/<recipient_domain>/mail/inbound/YYYY/MM/DD/<ingest_id>/edge.json`.
- Web-owned worker notification: a signed metadata-only hint that carries
  the existing org, Worker deployment, archive prefix, `ingest_id`, `raw_key`,
  `edge_key`, `result_key`, `recipient_domain`, `received_at`, and
  `raw_sha256`. It creates no additional durable message ID, does not carry raw
  mail bytes, and is verified by the web server before the internal
  `agentMail.ingest.enqueue` call wakes the same processing loop used by the R2
  sweep.
- Original `Message-ID`: copied from the inbound message when present.
- `X-ATM-Ingest-ID`: projected only during inbound replay into WildDuck.
- `X-ATMCF-*`: projected only during inbound replay into WildDuck. These are
  inbound edge provenance headers, not outbound provider headers.
- WildDuck user/mailbox/message IDs: resolved after replay and written to the
  inbound result as `wildduck_user_id`, `wildduck_mailbox_id`, and
  `wildduck_message_id`.
- Inbound DSN raw R2 key for permanent no-local-mailbox failures:
  `orgs/<org_public_id>/domains/<recipient_domain>/mail/inbound/YYYY/MM/DD/<ingest_id>/dsn.eml`.
- Inbound result R2 key:
  `orgs/<org_public_id>/domains/<recipient_domain>/mail/inbound/YYYY/MM/DD/<ingest_id>/result.json`.

The Mail Control API exposes a read-only inbound handoff operation,
`agentMail.message.provenance.get`, for callers that already have a WildDuck
delivery identity from mailbox updates. The operation fetches the exact
WildDuck stored source through
`/users/:user/mailboxes/:mailbox/messages/:uid/message.eml`, parses RFC822
headers through the service parser stack, and returns the canonical delivery
key used for downstream idempotency.

The canonical delivery key is:

```text
agent-mail:inbound:v1:ingest:<ingest>:wd:<user>:mb:<mailbox>:uid:<uid>
```

when `X-ATM-Ingest-ID` is present in the stored message. The idempotency basis
for that key is `ingest+wildduck-delivery`.

If the stored source is fetched but `X-ATM-Ingest-ID` is absent, the operation
must still return a deterministic fallback key. If the caller provided a true
WildDuck message ObjectId, the fallback key is:

```text
agent-mail:inbound:v1:wd-message:<message-object-id>:wd:<user>:mb:<mailbox>:uid:<uid>
```

with idempotency basis `wildduck-message+delivery`. Otherwise the fallback key
is:

```text
agent-mail:inbound:v1:wd:<user>:mb:<mailbox>:uid:<uid>
```

with idempotency basis `wildduck-delivery`.

The provenance API exposes only these header-derived fields:

- `X-ATM-Ingest-ID`
- `X-ATMCF-Edge-*`
- RFC `Message-ID`, as trace metadata only

It must not use the WildDuck update event ID, RFC `Message-ID`, or any
Cloudflare edge/message header as the canonical delivery key. If the WildDuck
source fetch fails, the API returns an error; consumers must retry and must not
create downstream work from incomplete provenance.

The Worker archive path must remain raw-byte exact. MIME parsing, header edits,
and body rewrites happen only after the raw object is durable.

WildDuck message source fetched through
`/users/:user/mailboxes/:mailbox/messages/:uid/message.eml` is the
mailbox-visible source used for display and header parsing. It is rebuilt from
WildDuck's MIME tree and must not be treated as the exact Cloudflare-boundary
raw byte stream. When raw-byte proof is required for inbound Worker mail, use
the R2 `raw.eml` object and its recorded `raw_sha256`.

The Control API message view/security surfaces build on the same WildDuck
delivery identity as `agentMail.message.provenance.get`:

- `agentMail.message.view.get`: returns plain text, sanitized display HTML,
  inert external-link markers plus external-link metadata, remote image
  metadata, and attachment-backed inline image metadata for authenticated
  frontend and operator display.
- `agentMail.message.security.get`: returns structured security evidence with
  explicit source labels and unavailable statuses.

These operations are read-only. They do not create a new canonical delivery key
and must use the same delivery key logic as `agentMail.message.provenance.get`.
They are authenticated Control API surfaces. Control API tokens must not be
projected to agent runtimes.
The view operation must block active remote image sources by default. Browser
renderers that request `remoteImages: allow` still must block those loads until
explicit user action. Links allowed for display must be rewritten to inert link
ID markers rather than returning original external `href` values inside
`displayHtml`.

Replay code must preserve every deactivated Cloudflare boundary authentication
or trace header as an inactive `X-ATMCF-Cloudflare-<Original-Header-Name>`
header in the replayed WildDuck copy. The exact original header lines remain
available in verified archived `raw.eml`; replay namespacing is only a final-copy
projection for display and diagnostics.

Current inbound security evidence has these limits:

- The Cloudflare Worker archive must persist every Worker-exposed email event
  property and header needed for security provenance in `edge.json`. This is a
  high-priority edge-evidence requirement, not optional UI metadata. The
  Control API must bind replay headers to the persisted manifest and Worker
  evidence for ingest ID, raw key, edge key, raw SHA-256, envelope recipient,
  envelope sender, and Worker received timestamp before trusting local
  Haraka/WildDuck authentication results.
- Cloudflare per-message SPF, DKIM, ARC, DMARC, BIMI, mailed-by, signed-by,
  and original peer-IP evidence must come from Cloudflare-added headers in the
  verified R2 `raw.eml`. The Control API must parse those headers with
  standards parsers after raw SHA-256 binding succeeds.
- The Worker must not query Cloudflare Analytics or GraphQL during inbound
  receive, and `edge.json` must not carry synthetic unavailable Cloudflare auth
  verdict markers.
- Downstream services must not infer Cloudflare evidence from Haraka replay,
  Rspamd scores, Worker request headers, or untrusted message headers in the
  WildDuck mailbox copy.
- Haraka/WildDuck SPF for Worker replay is internal replay SPF because replay
  enters Haraka from Agent Mail infrastructure, not from the original internet
  sender IP.
- WildDuck/Haraka DKIM, ARC, DMARC, and BIMI evidence must identify the Agent
  Mail boundary that generated it and must not be labeled as Cloudflare
  verdicts.
- Trusted Authentication-Results summaries require a topmost Agent Mail
  authserv boundary and verified R2 archive evidence for the Worker-ingested
  copy. Spoofed `X-ATM*` or `X-ATMCF*` headers in mailbox content are not enough
  to make local-looking auth results trusted.

The Haraka plugin set for this service is `rspamd` plus `wildduck`. The
WildDuck plugin already performs SPF, DKIM, ARC, DMARC, and BIMI through
mailauth. Provenance or security work must not add overlapping Haraka SPF or
DKIM plugins.

For DSNs with a null envelope sender, edge metadata stores `envelope_from` as
an empty string and replay provenance must project
`X-ATMCF-Edge-Envelope-From: <>`.

Inbound completion statuses preserve how the message left the local delivery
path:

- `delivered`: local WildDuck delivery was proven.
- `delivery_failed_dsn_submitted`: the local recipient did not exist or was
  permanently rejected, Agent Mail wrote `dsn.eml`, and ZoneMTA accepted that
  DSN through the dedicated DSN interface with internal `MAIL FROM:<>`.
- `delivery_failed_dsn_suppressed`: the local recipient failed permanently, but
  the original envelope sender was null or invalid, so Agent Mail suppressed the
  DSN to avoid backscatter.

For `delivery_failed_dsn_submitted`, the inbound result carries `dsn_id`,
`dsn_message_id`, `dsn_raw_key`, `dsn_raw_sha256`, `dsn_envelope_from`,
`dsn_envelope_to`, `dsn_from`, `dsn_status`, `dsn_action`,
and `dsn_diagnostic_code`. `dsn_id` is a UUIDv7 created by Agent Mail.
`dsn_envelope_from` is the internal null reverse path. Provider-bound DSN
behavior is recorded on the outbound DSN receipt because the internal SMTP
relay handles the selected provider and provider raw payload.

`dsn.eml` preserves provenance for the failed inbound message by carrying a
human-readable text/HTML report, populated `message/delivery-status` fields, and
the original archived message as `message/rfc822`.

## Outbound

Outbound mail flows through the canonical mail-server path:

- WildDuck `/users/:user/submit` is the authenticated agent/user submission
  surface.
- WildDuck native forwarding surfaces, including user `targets`, forwarded
  addresses, and forwarding filters, are outbound mail. They use the same
  ZoneMTA, internal SMTP relay, archive, and result provenance surfaces as other
  outbound messages.
- WildDuck writes the user-visible Sent copy through its normal submit path.
- ZoneMTA owns queueing, retry, bounce generation, and the SMTP handoff to the
  internal SMTP relay.
- The internal SMTP relay is a ZoneMTA-only SMTP hop. It archives the exact raw
  bytes received from ZoneMTA. For provider-bound recipients, it separately
  archives the sanitized provider-bound payload. For active local Agent Mail
  recipient domains, it records a local delivery result, stamps internal
  local-route provenance headers, and hands the exact local-delivery bytes back
  to Haraka/WildDuck without creating a provider payload.

The internal SMTP relay has two durable outbound copies for provider-bound
sends:

- Relay-received raw copy: the exact SMTP DATA bytes accepted from ZoneMTA.
- Provider-bound copy: provider-specific sanitized bytes or JSON payload. This
  is never written back over the relay-received raw copy or the WildDuck Sent
  copy.

Local active-domain routing has a source-side outbound archive and a target-side
inbound archive linked by `local_route_id`. The source-side outbound archive
stores the relay-received raw copy and records `provider: "local"` with
`local_routed` or `local_route_failed`. The target-side inbound archive stores
the delivered local mailbox copy after internal route headers are stamped, with
a local-route `edge.json`, and records `local_routed_delivered` or
`local_route_failed`. These records intentionally omit provider payload keys,
provider raw hashes, provider boundary sender, and provider message IDs because
no external provider handoff occurred.

Outbound surfaces:

- WildDuck submit response `message.mailbox`: mailbox ID for the stored Sent
  message.
- WildDuck submit response `message.id`: message ID for the stored Sent message.
- WildDuck submit response `message.queueId`: ZoneMTA queue ID for the outbound
  message.
- `X-Agent-Mail-ZoneMTA-Queue-ID`: internal ZoneMTA-to-relay header
  that binds the SMTP hop back to the WildDuck/ZoneMTA queue ID. It is stripped
  before any provider handoff.
- `send_id`: UUIDv7 generated by the internal SMTP relay for one outbound
  archive bundle. This is not the ZoneMTA queue ID.
- Relay raw R2 key:
  `mail/outbound/<source_domain>/YYYY/MM/DD/<send_id>/relay.eml`.
- Relay metadata R2 key:
  `mail/outbound/<source_domain>/YYYY/MM/DD/<send_id>/relay.json`.
- Original `Message-ID`: remains in the WildDuck Sent copy and relay raw
  archive.
- `message_id`: original `Message-ID` parsed from the relay-received message
  and copied into relay/result metadata when present.
- `is_dsn`, `dsn_id`, and `source_ingest_id`: set only for Agent
  Mail-generated DSNs. They bind the outbound DSN bundle back to the inbound
  failure that caused it.
- `provider_raw_sha256`: SHA-256 of the SES provider-bound sanitized raw copy
  or Cloudflare Email Sending JSON payload.
- Provider-bound R2 key for provider sends: either
  `mail/outbound/<source_domain>/YYYY/MM/DD/<send_id>/provider.eml` for SES raw
  RFC822 or `mail/outbound/<source_domain>/YYYY/MM/DD/<send_id>/provider.json`
  for Cloudflare Email Sending JSON.
- `provider_message_id`: provider-returned ID, such as SES `MessageId`.
- `provider_boundary_sender`: the sender address Agent Mail had to present at
  the provider API boundary for feedback routing.
- `provider_reverse_path_mode`: the policy used for the provider boundary, such
  as `configured_return_path`, `cloudflare_json_from`, or
  `provider_feedback_fallback`.
- Outbound result R2 key:
  `mail/outbound/<source_domain>/YYYY/MM/DD/<send_id>/result.json`.

Provider-bound outbound messages must not carry inbound or internal headers:
`X-ATMCF-*`, `X-ATM-Ingest-ID`, `Bcc`, `Return-Path`, `Received`,
`DKIM-Signature`, `Authentication-Results`, `ARC-*`,
`X-Agent-Mail-ZoneMTA-Queue-ID`, Rspamd, Haraka, spam, or SES authorization
headers.

For SES, the provider-bound raw copy gets exactly one configured
`Return-Path: <bounces@sender-domain>`, sourced from
the active sender-domain registry entry.

For Agent Mail-generated DSNs, ZoneMTA submits to the internal SMTP relay with
an empty SMTP envelope sender. The relay preserves that internal null intent in
`relay.json`/`result.json`, strips internal DSN correlation headers from the
provider-bound payload, and records `source_ingest_id`, `dsn_id`, and
`provider_reverse_path_mode`. SES DSN sends use
`provider_reverse_path_mode: "provider_feedback_fallback"` and provider raw
`Return-Path: <bounces@sender-domain>` because SES `SendRawEmail` rejects raw
`Return-Path: <>` as an empty address. Providers that cannot request a null
sender use the per-domain feedback mailbox fallback and record
`provider_reverse_path_mode: "api_from_fallback"`.

For forwarded copies to active local Agent Mail domains, the visible `From`
domain may belong to a domain Agent Mail does not own. That visible sender
domain is metadata only and must not be the outbound archive path. The outbound
archive path for a local routed copy uses the source mailbox domain, and the
target-side inbound archive path uses the target mailbox domain. Provider
sender-domain policy is not consulted for these local deliveries;
recipient-domain active state decides whether the relay should deliver locally.
If the relay-received message carries `X-ATMCF-Edge-Envelope-From`, local
delivery to Haraka uses that original replay envelope sender as SMTP
`MAIL FROM`. ZoneMTA's SRS-expanded reverse path is preserved in relay
metadata, but it is not the local target-mailbox delivery envelope.

Target-side local routed inbound records use `edge.json` explicitly with
`schema: "agent-mail.inbound.local-route.edge.v1"`. This edge is not a
Cloudflare Worker commit marker and must not be replayed by the reconciler. It
is the local routing boundary record that links the target mailbox copy to
`local_route_id`, the source mailbox, the ZoneMTA queue ID, and the source
inbound archive when a source ingest ID is available.

The internal SMTP relay must stamp local routed deliveries with
`X-Agent-Mail-Local-Route-ID`, `X-Agent-Mail-Source-Mailbox`,
`X-Agent-Mail-Target-Mailbox`, and `X-Agent-Mail-Source-Ingest-ID` when a source
ingest ID is available. Target-side WildDuck proof must be queryable by
`X-Agent-Mail-Local-Route-ID` and target mailbox identity.

If ZoneMTA retries after a local mailbox delivery succeeded but route result
persistence failed, the internal SMTP relay must recover the target mailbox
copy by ZoneMTA queue ID, source ingest ID when available, parsed `Message-ID`,
and target mailbox identity. When the stored copy has
`X-Agent-Mail-Local-Route-ID`, the internal SMTP relay must reuse that route ID
and finish the missing R2 records instead of delivering another mailbox copy.
