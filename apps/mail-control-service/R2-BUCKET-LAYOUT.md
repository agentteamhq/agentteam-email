# Agent Mail R2 Bucket Layout

This document is the required bucket layout contract for the Agent Mail archive
bucket. R2 is the immutable mail artifact store. It is not the live queue, retry
store, or delivery database.

## Rules

- Every Agent Mail-created object identity is a UUIDv7 in canonical lowercase
  string form. This includes `ingest_id`, `send_id`, and any future attempt or
  event IDs. External IDs such as WildDuck message IDs, ZoneMTA queue IDs, and
  provider message IDs are recorded as metadata, not reused as Agent Mail object
  identities.
- `raw_sha256` and `provider_raw_sha256` are integrity fields only. They must
  never be used as object IDs.
- Worker-origin inbound artifacts are grouped by organization public ID,
  canonical recipient domain, direction, UTC date, UUIDv7 ID, then file:
  `orgs/<org_public_id>/domains/<recipient_domain>/mail/inbound/...`. Do not
  reintroduce top-level `queue/`, `receipt/`, or `deadletter/` prefixes.
- Domain and subdomain path segments are parser-derived canonical Agent Mail
  mailbox domains. Inbound uses the recipient mailbox domain. Provider-bound
  outbound uses the active provider sender domain. Local routed outbound uses
  the source mailbox domain. A visible `From` domain that Agent Mail does not
  own is metadata only and must never be the archive root for a local routed
  copy.
- The concrete routed or sender address and the selected domain must also be
  recorded in metadata. The path is an index for listing and reconciliation; it
  is not the only source of domain truth.
- Production code must use a shared Agent Mail key builder/parser for this
  layout. Do not scatter ad hoc string concatenation or path splitting across
  Worker, reconciler, relay, tests, or admin tooling.
- All date path segments and timestamps are UTC. Calendar arithmetic must use
  standard date/time APIs, not string math, integer day increments, or the
  process/system timezone.
- Files are state-dependent, not optional. If a message reaches a lifecycle
  state that requires an artifact, that artifact is mandatory and tested.

## Inbound

Inbound IDs are generated by the Cloudflare Email Worker as UUIDv7 values.
`ingest_id` identifies one inbound Cloudflare-boundary message archive bundle.
It is not derived from the raw message hash, mailbox, `Message-ID`, or
Cloudflare routing data.

Required inbound bundle path:

```text
orgs/<org_public_id>/domains/<recipient_domain>/mail/inbound/YYYY/MM/DD/<ingest_id>/
  raw.eml
  edge.json
  dsn.eml
  result.json
```

`raw.eml` is the exact raw RFC822 bytes received from Cloudflare. The Worker
must not parse MIME, edit headers, rewrite bodies, or inject provenance headers
before this object is written.

For Worker-origin inbound bundles, `edge.json` is the Worker boundary metadata
and inbound commit marker. It is written after `raw.eml` succeeds. The
reconciler discovers inbound edge objects by listing the active domain's
service-owned archive prefix for objects ending in `/edge.json`, then
classifies them by schema.

After `edge.json` is committed, the Worker must attempt one HMAC-signed
fast-path notification through the configured web-owned ingest URL. The
notification is not an R2 artifact, does not carry raw mail bytes, and does not
define completion. The web server verifies the Worker request and calls
`agentMail.ingest.enqueue`; mail-control then validates the metadata against
active control state before inserting the committed bundle into Mongo state and
waking the same processing loop used by the six-hour sweep.

Required `edge.json` fields:

```json
{
  "schema": "agent-mail.inbound.edge.v1",
  "ingest_id": "018f0000-0000-7000-8000-000000000000",
  "raw_key": "orgs/<org_public_id>/domains/<recipient_domain>/mail/inbound/YYYY/MM/DD/<ingest_id>/raw.eml",
  "raw_sha256": "<sha256-of-raw-eml>",
  "mailbox": "agent@example.com",
  "envelope_from": "",
  "envelope_to": "agent@example.com",
  "recipient_domain": "example.com",
  "cloudflare_zone_name": "example.com",
  "worker_name": "agent-mail-worker",
  "received_at": "YYYY-MM-DDTHH:MM:SSZ",
  "message_id": "<original-message-id-when-present>",
  "atmcf_headers": {},
  "cloudflare_edge_evidence": {}
}
```

`cloudflare_edge_evidence` is the Worker-observed receive surface: envelope
fields, raw size, Worker timestamp, and the `message.headers.entries()` header
snapshot. It must not contain synthetic unavailable records for auth verdicts.
Cloudflare auth verdict evidence is parsed downstream from the verified R2
`raw.eml` Cloudflare-added headers after `edge.json` binds the archive key,
envelope, timestamp, and raw SHA-256. The Worker does not query Cloudflare
Analytics or GraphQL during the receive path.

Local routed inbound records also use `edge.json`, but with a different schema.
They are explicit archive/provenance markers for mail already delivered locally
by the internal SMTP relay, not Worker replay work.

Required local routed inbound bundle path:

```text
mail/inbound/<target_domain>/YYYY/MM/DD/<local_route_id>/
  raw.eml
  edge.json
  result.json
```

Required local routed `edge.json` fields:

```json
{
  "schema": "agent-mail.inbound.local-route.edge.v1",
  "local_route_id": "018f0000-0000-7000-8000-000000000000",
  "raw_key": "mail/inbound/<target_domain>/YYYY/MM/DD/<local_route_id>/raw.eml",
  "raw_sha256": "<sha256-of-raw-eml>",
  "source_mailbox": "media@example.com",
  "source_domain": "example.com",
  "target_mailbox": "agent@example.com",
  "target_domain": "example.com",
  "visible_from": "Sender <sender@example.net>",
  "visible_sender_domain": "example.net",
  "zonemta_queue_id": "...",
  "source_ingest_id": "...",
  "source_outbound_relay_key": "mail/outbound/<source_domain>/YYYY/MM/DD/<local_route_id>/relay.eml",
  "source_outbound_result_key": "mail/outbound/<source_domain>/YYYY/MM/DD/<local_route_id>/result.json",
  "routed_at": "YYYY-MM-DDTHH:MM:SSZ"
}
```

The reconciler must enqueue only `schema: "agent-mail.inbound.edge.v1"` Worker
edges into the Mongo-backed queue. It must recognize
`schema: "agent-mail.inbound.local-route.edge.v1"` as local routed archive state
and must not replay or enqueue it.

`envelope_from` is an empty string for DSNs with a null envelope sender. Replay
must project `X-ATMCF-Edge-Envelope-From: <>`, but the raw archive remains
unchanged.

`dsn.eml` is required only for `delivery_failed_dsn_submitted`. It is the
Agent Mail-generated unknown-recipient DSN submitted to ZoneMTA with a null SMTP
reverse path. It includes human-readable text and HTML alternatives, a populated
`message/delivery-status` part, and the original archived message as
`message/rfc822` for sender/client correlation. It is not written for delivered
mail or `delivery_failed_dsn_suppressed`.

`result.json` is the inbound completion marker for the bundle. It is written
only after delivery has been proven in WildDuck, after the consumer confirms
the message was already delivered by an earlier attempt, after a DSN has been
submitted for a permanent no-local-mailbox failure, or after DSN generation is
suppressed because the original sender cannot receive a DSN. It is the current
completion marker for inbound bundles.

Required `result.json` fields:

```json
{
  "schema": "agent-mail.inbound.result.v1",
  "ingest_id": "018f0000-0000-7000-8000-000000000000",
  "status": "delivered",
  "attempt": 1,
  "processed_at": "YYYY-MM-DDTHH:MM:SSZ",
  "raw_key": "orgs/<org_public_id>/domains/<recipient_domain>/mail/inbound/YYYY/MM/DD/<ingest_id>/raw.eml",
  "edge_key": "orgs/<org_public_id>/domains/<recipient_domain>/mail/inbound/YYYY/MM/DD/<ingest_id>/edge.json",
  "wildduck_user_id": "...",
  "wildduck_mailbox_id": "...",
  "wildduck_message_id": "...",
  "delivery_source": "replayed"
}
```

For a submitted DSN, `result.json` uses the same base fields and adds DSN
provenance:

```json
{
  "schema": "agent-mail.inbound.result.v1",
  "ingest_id": "018f0000-0000-7000-8000-000000000000",
  "status": "delivery_failed_dsn_submitted",
  "attempt": 1,
  "processed_at": "YYYY-MM-DDTHH:MM:SSZ",
  "raw_key": "orgs/<org_public_id>/domains/<recipient_domain>/mail/inbound/YYYY/MM/DD/<ingest_id>/raw.eml",
  "edge_key": "orgs/<org_public_id>/domains/<recipient_domain>/mail/inbound/YYYY/MM/DD/<ingest_id>/edge.json",
  "dsn_raw_key": "orgs/<org_public_id>/domains/<recipient_domain>/mail/inbound/YYYY/MM/DD/<ingest_id>/dsn.eml",
  "dsn_raw_sha256": "<sha256-of-dsn-eml>",
  "dsn_id": "018f0000-0000-7000-8000-000000000001",
  "dsn_message_id": "<018f0000-0000-7000-8000-000000000001@recipient-domain>",
  "dsn_envelope_from": "",
  "dsn_envelope_to": "original-sender@example.net",
  "dsn_from": "bounces@<recipient_domain>",
  "dsn_status": "5.1.1",
  "dsn_action": "failed",
  "dsn_diagnostic_code": "smtp; 550 5.1.1 No such user",
  "delivery_source": "dsn_submitted"
}
```

For a suppressed DSN, `result.json` uses status
`delivery_failed_dsn_suppressed`, records the DSN status/action/diagnostic,
sets `delivery_source: "dsn_suppressed"`, and records the suppression reason in
`detail`. It does not create `dsn.eml`.

Valid inbound completion statuses are `delivered`,
`delivery_failed_dsn_submitted`, and `delivery_failed_dsn_suppressed`. If replay
or DSN submission fails transiently, the bundle remains incomplete and
`result.json` is not written. The reconciler makes one initial attempt plus
exactly two retries. After the third failed processing attempt, the Mongo queue item
becomes `blocked` until an operator fixes the underlying cause and resets the
queue item to `pending`. Blocked items are not Agent Mail message terminal state and
must not create R2 deadletter objects.

## Outbound

Outbound `send_id` values are generated by the internal SMTP relay as UUIDv7
values. `send_id` identifies one outbound relay archive bundle. It is not the
ZoneMTA queue ID and must not be derived from the ZoneMTA queue ID.

Required outbound bundle path:

```text
mail/outbound/<source_domain>/YYYY/MM/DD/<send_id>/
  relay.eml
  relay.json
  provider.eml | provider.json
  result.json
```

`provider.eml` or `provider.json` is required only for provider-bound
deliveries. Local active-domain deliveries do not create a provider payload
object because the exact local-delivery payload is archived as the target-side
inbound `raw.eml`.

`relay.eml` is the exact SMTP DATA bytes accepted from ZoneMTA. It preserves the
internal `X-Agent-Mail-ZoneMTA-Queue-ID` header.

`relay.json` is the internal SMTP relay boundary metadata.

Required `relay.json` fields:

```json
{
  "schema": "agent-mail.outbound.relay.v1",
  "send_id": "018f0000-0000-7000-8000-000000000000",
  "route_type": "provider",
  "relay_raw_key": "mail/outbound/<source_domain>/YYYY/MM/DD/<send_id>/relay.eml",
  "relay_raw_sha256": "<sha256-of-relay-eml>",
  "zonemta_queue_id": "...",
  "wildduck_user_id": "...",
  "wildduck_mailbox_id": "...",
  "wildduck_message_id": "...",
  "envelope_from": "agent@example.com",
  "envelope_to": ["recipient@example.net"],
  "sender_domain": "example.com",
  "source_domain": "example.com",
  "delivery_domain": "example.com",
  "provider": "ses",
  "submitted_at": "YYYY-MM-DDTHH:MM:SSZ"
}
```

For provider-bound deliveries, `provider` is the selected outbound provider.
For active local-domain deliveries, `route_type` is `local`, `provider` is
`local`, and `send_id` is the same value as `local_route_id`.

Exactly one provider-bound payload object is required once a provider payload is
built:

- `provider.eml` for SES raw RFC822 handoff.
- `provider.json` for Cloudflare Email Sending JSON handoff.

The provider-bound payload must remove inbound and internal headers, including
`X-ATMCF-*`, `X-ATM-Ingest-ID`, `X-Agent-Mail-ZoneMTA-Queue-ID`, `Bcc`,
`X-Agent-Mail-Local-Route-ID`, `X-Agent-Mail-Source-Mailbox`,
`X-Agent-Mail-Target-Mailbox`, `X-Agent-Mail-Source-Ingest-ID`, `Return-Path`,
`Received`, DKIM, Authentication-Results, ARC, Rspamd, Haraka, spam, and SES
authorization headers. The relay archive is not rewritten.

`result.json` is the terminal outbound result for the bundle. It is the current
completion marker for outbound relay bundles.

Required `result.json` fields:

```json
{
  "schema": "agent-mail.outbound.result.v1",
  "send_id": "018f0000-0000-7000-8000-000000000000",
  "status": "provider_accepted",
  "provider": "ses",
  "provider_message_id": "...",
  "provider_payload_key": "mail/outbound/<source_domain>/YYYY/MM/DD/<send_id>/provider.eml",
  "provider_raw_sha256": "<sha256-of-provider-payload>",
  "provider_boundary_sender": "<provider-boundary-sender>",
  "provider_reverse_path_mode": "<provider-reverse-path-policy>",
  "delivered": [],
  "queued": ["recipient@example.net"],
  "error": "",
  "completed_at": "YYYY-MM-DDTHH:MM:SSZ"
}
```

For Agent Mail-generated DSNs sent through SES,
`provider_reverse_path_mode` is `provider_feedback_fallback`,
`provider_boundary_sender` is the service-owned feedback mailbox, and
`provider.eml` contains sanitized `Return-Path: <bounces@sender-domain>`.
For providers that cannot request a null sender, `provider_reverse_path_mode`
is `api_from_fallback` and `provider_boundary_sender` is the service-owned
feedback mailbox.

Valid outbound terminal statuses are `provider_accepted`,
`provider_rejected`, `provider_failed`, `local_routed`, and
`local_route_failed`. ZoneMTA retry state remains ZoneMTA state; Agent Mail
records every outbound relay bundle it accepts. Local routing results use
`provider: "local"`, link to the target-side inbound route result, and omit
provider payload fields.

For local routing, the internal SMTP relay must stamp the local delivery
payload with `X-Agent-Mail-Local-Route-ID`, `X-Agent-Mail-Source-Mailbox`,
`X-Agent-Mail-Target-Mailbox`, and `X-Agent-Mail-Source-Ingest-ID` when a
source ingest ID is available. The target-side inbound `raw.eml` stores those
exact local-delivery bytes. Source-side outbound `relay.eml` remains the exact
ZoneMTA-to-relay bytes.
When `relay.eml` contains `X-ATMCF-Edge-Envelope-From`, the internal SMTP relay
must use that original replay envelope sender for the local Haraka `MAIL FROM`.
ZoneMTA's SRS-expanded reverse path remains in relay/result metadata but must
not be used as the local target-mailbox delivery envelope.

If a retry finds a previously delivered target mailbox copy with
`X-Agent-Mail-Local-Route-ID`, the internal SMTP relay must reuse that
`local_route_id` for any missing local-route R2 records rather than creating a
new route ID.

## Order of Operations

Inbound:

1. Worker receives a Cloudflare Email Routing message.
2. Worker parses the envelope recipient address and derives canonical
   `recipient_domain`.
3. Worker resolves the authenticated web-owned domain deployment to
   `organization_id`, `organization_public_id`, `archive_prefix`,
   `worker_connection_id`, and `worker_domain_deployment_id`.
4. Worker generates UUIDv7 `ingest_id` and computes `raw_sha256`.
5. Worker writes `raw.eml`.
6. Worker writes `edge.json`; this is the durable inbound commit marker.
7. Worker sends one HMAC-signed fast-path notification to the web-owned ingest
   boundary containing `organization_id`, `organization_public_id`,
   `archive_prefix`, `worker_connection_id`, `worker_domain_deployment_id`,
   `ingest_id`, `recipient_domain`, `raw_key`, `edge_key`, `result_key`,
   `received_at`, and `raw_sha256`. Notification failure is logged and not
   retried by the Worker. A failed notification does not change the R2 archive
   state.
8. The web-owned ingest boundary verifies the Worker connection and calls
   `agentMail.ingest.enqueue` on the internal control API. The control service
   validates the org, archive prefix, domain, and Worker deployment metadata
   against active control state, then upserts the bundle into Mongo as
   `pending` and wakes the replay processing loop.
9. The reconciler sweep also discovers `edge.json`, classifies the edge schema,
   and checks for sibling `result.json`.
10. If `result.json` exists, the bundle is complete and no body object is read.
11. The reconciler reads `edge.json` and `raw.eml`, verifies `raw_sha256`, checks
    WildDuck for an existing `X-ATM-Ingest-ID`, then replays if needed.
12. After WildDuck delivery is proven, the reconciler writes `result.json` with
    status `delivered`.
13. If WildDuck address resolution or Haraka replay returns a permanent
    no-local-mailbox failure, the reconciler stores DSN state in Mongo,
    writes `dsn.eml`, submits it to ZoneMTA with `MAIL FROM:<>`, then writes
    `result.json` with status `delivery_failed_dsn_submitted`.
14. If the original envelope sender is null or invalid, the reconciler writes
    `result.json` with status `delivery_failed_dsn_suppressed` and does not
    create `dsn.eml`.

Inbound retry:

1. Any retryable failure before `result.json`, including DSN R2 writes or
   ZoneMTA submission, keeps the bundle incomplete.
2. The durable Mongo work state records the failure, attempt count, last
   error, next retry time, and whether the item is blocked.
3. The sweeper rediscovers incomplete Worker bundles by listing `edge.json`
   objects, classifying schema, and checking for sibling `result.json`.
4. The sweeper does not need to read `raw.eml` for bundles that already have
   `result.json`. It may need to read `edge.json` metadata to distinguish
   Worker replay work from local routed archive state.

Outbound:

1. WildDuck accepts user submission or native forwarding and ZoneMTA queues the
   message.
2. ZoneMTA submits to the internal SMTP relay with
   `X-Agent-Mail-ZoneMTA-Queue-ID`.
3. Provider-bound outbound derives canonical `sender_domain` from active
   provider policy. Local routed outbound derives the archive domain from the
   source mailbox domain, not the visible `From` domain.
4. The internal SMTP relay generates UUIDv7 `send_id` and writes `relay.eml`.
5. The internal SMTP relay writes `relay.json`.
6. If every envelope recipient belongs to an active local Agent Mail domain,
   the internal SMTP relay writes source-side outbound local-route metadata,
   stamps internal local-route provenance headers, delivers those exact
   local-delivery bytes to Haraka/WildDuck, writes target-side inbound
   `raw.eml`, local-route `edge.json`, and `result.json`, then writes the
   source-side outbound `result.json`.
7. Otherwise, the internal SMTP relay builds and writes the provider-bound
   payload as `provider.eml` or `provider.json`.
8. The internal SMTP relay sends the payload to the provider.
9. The internal SMTP relay writes `result.json` with the provider outcome.

Outbound retry:

- ZoneMTA owns outbound retry and bounce behavior. If ZoneMTA retries by
  submitting again to the internal SMTP relay, the relay creates a new
  UUIDv7 `send_id` and a new outbound bundle. The shared `zonemta_queue_id`
  links those bundles.
- The daily sweeper is allowed to detect outbound bundles missing
  `result.json`, but it must not re-send provider payloads from R2. Provider
  handoff is not idempotent from R2 alone.

## Operational State Store

R2 is the durable archive and completion marker store. The service-owned
`agent_mail_control` MongoDB database is the durable operational state store
for reconciliation, retries, leases, blocked items, DSN state, discovery
diagnostics, and sweep cursors.

The MongoDB state database is required. It must live as a separate database on
the shared Agent Mail MongoDB server and must not share WildDuck collections.
Do not use Kubernetes ConfigMaps, R2 object metadata, local temporary files,
SQLite files, or in-memory maps for retry state, blocked-item state, or sweep
cursors.

The control service reconciler is the only component that leases and processes
work items. The internal `agentMail.ingest.enqueue` handler only upserts
committed inbound bundles into the same Mongo queue and wakes the normal
due-work loop.

Required inbound work item fields:

```text
ingest_id
recipient_domain
utc_date
bundle_prefix
edge_key
raw_key
result_key
status
attempt_count
first_seen_at
last_seen_at
first_attempt_at
last_attempt_at
next_attempt_at
lease_id
lease_until
failure_class
last_error
dsn_id
dsn_message_id
dsn_raw_key
blocked_at
created_at
updated_at
```

`ingest_id` is the primary key. `edge_key` is unique. Queue leasing and state
transitions must use MongoDB atomic update operations.

Valid inbound queue statuses:

- `pending`: discovered and eligible for processing.
- `leased`: one worker owns the item until `lease_until`.
- `retry_wait`: failed previously and must not retry before
  `next_attempt_at`.
- `blocked`: exceeded the configured retry policy or hit an invariant failure.
  This is the replacement for a message-level dead letter. It is not terminal
  archive state because `result.json` is still missing.
- `delivered`: this reconciler wrote a delivered `result.json`.
- `completed`: `result.json` exists and the row no longer needs work. This
  covers already-completed discoveries and DSN terminal results without reading
  or reclassifying the result body.

Required sweep cursor fields:

```text
direction
canonical_domain
last_sweep_end_at
last_success_at
last_error
created_at
updated_at
```

Required discovery diagnostic fields for malformed or unrecoverable objects:

```text
object_key
canonical_domain
first_seen_at
last_seen_at
failure_class
last_error
created_at
updated_at
```

The Mongo collections, indexes, and validators must be created and migrated by
the Mail Control Service using
`github.com/xakep666/mongo-migrate`.
Missing or incompatible schema is a fatal startup error unless an explicit
checked-in migration is run successfully.

## Processing Contract

The reconciler loop processes due Mongo queue items. It must lease one item at
a time with an atomic `FindOneAndUpdate` so one item failure cannot block later
items.

Processing order for each due row:

1. Lease one eligible queue item:
   `status in (pending, retry_wait)`, `next_attempt_at <= now`, and no active
   lease.
2. Commit the lease with a UUIDv7 `lease_id` and bounded `lease_until`.
3. `HEAD` sibling `result.json`. If it exists, mark queue status `completed`
   and stop.
4. Read and validate `edge.json`.
5. Read `raw.eml` and verify `raw_sha256`.
6. Check WildDuck for an existing delivered message with `X-ATM-Ingest-ID`.
7. If WildDuck already contains the message, write `result.json` with
   `delivery_source: "existing"` and mark queue status `delivered`.
8. If WildDuck reports the local recipient does not exist, handle the permanent
   delivery failure through the DSN path.
9. If WildDuck does not contain the message, replay through the configured
   inbound SMTP path.
10. If SMTP replay returns a permanent local recipient failure, handle it
    through the DSN path.
11. Wait for WildDuck user, mailbox, and message IDs.
12. Write `result.json` with `delivery_source: "replayed"`.
13. Mark queue status `delivered`.

DSN path processing order:

1. If the original envelope sender is null or invalid, write
   `delivery_failed_dsn_suppressed` and mark queue status `completed`.
2. Resolve the per-domain feedback mailbox from the active domain registry
   projection.
3. In Mongo state, persist or reuse `dsn_id`, `dsn_message_id`, and
   `dsn_raw_key` for the original `ingest_id`.
4. If `dsn.eml` already exists, reuse it. Otherwise build the DSN MIME message
   with human-readable text and HTML alternatives, populated delivery-status
   fields, and the original archived message as `message/rfc822`, then write it
   to `dsn_raw_key`.
5. Submit the DSN to ZoneMTA's dedicated DSN interface with an empty SMTP
   envelope sender and the original envelope sender as the only recipient.
6. Write `result.json` with `delivery_failed_dsn_submitted` and mark queue
   status `completed`.

Processing failures:

- A retryable failure on attempt 1 or 2 updates the row to `retry_wait`,
  increments `attempt_count`, stores `last_error`, clears the lease, and sets
  `next_attempt_at` from the checked-in retry policy.
- A retryable failure on attempt 3 updates the row to `blocked`.
- An invariant failure or exhausted retry policy updates the row to `blocked`,
  stores `last_error`, clears the lease, and sets `blocked_at`.
- The reconciler must continue with other due rows after any item failure.
- Blocked rows are operator-actionable reconciler state. They do not create R2
  deadletter objects and do not write `result.json`.
- Every processing failure must log structured service output visible in the
  deployment logs, including `ingest_id`, domain, bundle prefix, current operation,
  attempt count, failure class, next retry or blocked state, and sanitized
  error detail.
- After the underlying cause is fixed, operator recovery resets a blocked row
  to `pending`. The next reconciler pass either delivers it or confirms that
  `result.json` already exists.

Required failure classes:

- `transient`: expected to recover without code or configuration changes, such
  as temporary WildDuck, Haraka, Mongo, Redis, network, or R2 errors.
- `dependency`: a required service state is unavailable, such as WildDuck,
  Haraka, ZoneMTA, Mongo, Redis, or R2 being unreachable. A confirmed missing
  local mailbox is not a retryable dependency failure; it is a permanent
  delivery failure handled through the DSN path.
- `invariant`: archive or metadata is internally inconsistent, such as missing
  `raw.eml`, `raw_sha256` mismatch, malformed `edge.json`, invalid UUIDv7, or
  path/domain mismatch.
- `bug`: parser, serializer, replay, or provider integration failed in a way
  that requires code change.

Retry policy values, including maximum retries, backoff schedule, lease
duration, sweep safety lag, and sweep overlap, must come from explicit
repo-managed configuration. `max_retries` must be `2`. Missing values are
fatal.

## Sweep Contract

Bucket listing is the reconciliation path for committed inbound bundles. It
must not block on individual message failures.

The sweeper runs every six hours in production. Test fixtures may use shorter
explicitly configured intervals, but production configuration must remain six
hours. The sweeper tracks a checkpoint for each `direction + canonical_domain`
in Mongo. It uses a lagged sliding window:

```text
sweep_end = now_utc - safety_lag
sweep_start = previous_last_sweep_end_at - overlap
```

If no cursor exists for a domain, `sweep_start` comes from explicit
repo-managed configuration such as `archive_start_at`.

Sweep order:

1. Load configured Agent Mail domains.
2. For each domain, compute `sweep_start` and `sweep_end` in UTC.
3. Expand every UTC calendar date touched by the window using standard
   date/time APIs.
4. List
   `orgs/<org_public_id>/domains/<domain>/mail/inbound/YYYY/MM/DD/` from the
   active domain's `archive_prefix` and select keys ending in `/edge.json`.
5. Ignore keys whose UUIDv7 timestamp falls outside the sweep window.
6. For each eligible `edge.json`, derive sibling `result.json` and issue a
   `HEAD`.
7. If `result.json` exists, upsert queue status `completed`.
8. If `result.json` is missing, upsert or refresh a Mongo queue item without
   changing `leased`, `retry_wait`, or `blocked` rows.
9. Continue listing after every per-item error. Record invalid keys or
   unrecoverable discovery errors in Mongo diagnostics and continue.
10. Advance `last_sweep_end_at` only after the domain scan completes.

The overlap is intentional. Re-listing already completed bundles is cheap
because `result.json` is the idempotency marker and Mongo records completed
rows. The sweeper must never advance by "last successful message"; it advances
by completed UTC time window so poisoned items cannot stall reconciliation.

## Analytics

Iceberg, DuckLake, or another table format is allowed only as a derived
analytics or audit index over these bundle paths. It must not be part of the
Worker archive commit path or the live delivery queue.
